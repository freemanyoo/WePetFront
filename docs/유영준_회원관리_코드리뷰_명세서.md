# 유영준 (회원관리) - 코드 리뷰 명세서

## 개요
본 문서는 `findmyfet` 프로젝트의 사용자 인증 및 회원 관리 시스템에 대한 코드 리뷰 명세서입니다. 유영준 님이 담당하신 사용자 인증, 회원가입/로그인 API, JWT 인증, 회원 관리 UI 기능을 중심으로 작성되었습니다.

## 1. 백엔드 (Spring Boot)

### 1.1. `User.java` (도메인 엔티티)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\domain\User.java`
- **주요 기능**:
    - 사용자 정보를 데이터베이스에 매핑하는 JPA 엔티티입니다.
    - `userId` (Primary Key), `loginId`, `password`, `name`, `phoneNumber`, `email`, `address`, `role`, `refreshToken` 필드를 포함합니다.
    - `loginId`와 `email`은 `unique = true`로 설정되어 중복을 방지합니다.
    - `BaseEntity`를 상속받아 생성 및 수정 시간을 자동으로 관리합니다.
    - `refreshToken` 필드를 통해 JWT 리프레시 토큰을 저장하고 관리합니다.
    - `@JsonIgnore` 어노테이션을 사용하여 `posts` 필드의 순환 참조 및 프록시 직렬화를 방지합니다.
- **코드 리뷰 포인트**:
    - `phoneNumber`와 `address` 필드에 대한 유효성 검사(예: 형식, 길이)가 DTO 또는 서비스 계층에서 추가적으로 필요할 수 있습니다.
    - `password` 필드는 암호화되어 저장되므로, 엔티티 자체에서 직접 접근하여 사용하지 않도록 주의해야 합니다.

### 1.2. `Role.java` (Enum)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\domain\Role.java`
- **주요 기능**:
    - 사용자 역할을 `USER`와 `ADMIN`으로 정의하는 Enum입니다.
- **코드 리뷰 포인트**:
    - 역할이 추가될 경우 Enum에 새로운 값을 추가하고, 관련 로직(예: 권한 부여)을 업데이트해야 합니다.

### 1.3. `UserRepository.java` (리포지토리)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\repository\UserRepository.java`
- **주요 기능**:
    - `User` 엔티티에 대한 데이터베이스 접근을 담당하는 Spring Data JPA 리포지토리입니다.
    - `findByLoginId(String loginId)`: 로그인 ID로 사용자를 조회합니다.
    - `findByEmail(String email)`: 이메일로 사용자를 조회합니다.
- **코드 리뷰 포인트**:
    - 복잡한 쿼리가 필요할 경우 `@Query` 어노테이션을 사용하거나 QueryDSL과 같은 쿼리 빌더를 고려할 수 있습니다.

### 1.4. `UserService.java` (서비스)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\service\UserService.java`
- **주요 기능**:
    - **`signup(UserSignupRequestDTO requestDTO)`**:
        - 회원가입 로직을 처리합니다.
        - `loginId`와 `email`의 중복 여부를 확인하여 예외를 발생시킵니다.
        - 비밀번호를 `BCryptPasswordEncoder`로 암호화하여 저장합니다.
        - 새로운 `User` 객체를 생성하고 데이터베이스에 저장합니다.
    - **`login(String loginId, String password)`**:
        - 사용자 로그인 로직을 처리합니다.
        - `loginId`로 사용자를 찾고, 비밀번호 일치 여부를 확인합니다.
        - `JwtUtil`을 사용하여 `accessToken`과 `refreshToken`을 생성합니다.
        - 생성된 `refreshToken`을 사용자 엔티티에 업데이트하고 저장합니다.
        - `LoginResponseDTO`를 반환하여 토큰과 사용자 정보를 프런트엔드에 전달합니다.
    - **`getMe(Long userId)`**:
        - 주어진 `userId`로 사용자 정보를 조회하여 `UserInfoResponseDTO`로 반환합니다.
    - **`updateMe(Long userId, UserUpdateRequestDTO requestDTO)`**:
        - 사용자 프로필 정보를 업데이트합니다.
        - 이름, 전화번호, 주소, 이메일 필드를 업데이트할 수 있습니다.
        - 비밀번호 변경 요청이 있을 경우, 현재 비밀번호를 확인하고 새 비밀번호를 암호화하여 업데이트합니다.
    - **`refreshAccessToken(String refreshToken)`**:
        - `refreshToken`의 유효성을 검사하고, 새로운 `accessToken`을 발급합니다.
        - 저장된 `refreshToken`과 일치하는지 확인하여 보안을 강화합니다.
- **코드 리뷰 포인트**:
    - `signup` 및 `login` 메서드에서 발생하는 `IllegalArgumentException`에 대해 더 구체적인 커스텀 예외를 정의하여 예외 처리를 세분화할 수 있습니다.
    - `updateMe`에서 이메일 변경 시 중복 이메일 검증 로직이 추가되면 좋습니다.
    - 비밀번호 변경 로직에서 새 비밀번호의 복잡성(예: 특수문자, 숫자 포함)을 검증하는 로직을 추가할 수 있습니다.

### 1.5. `UserController.java` (컨트롤러)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\controller\UserController.java`
- **주요 기능**:
    - RESTful API 엔드포인트를 정의합니다.
    - `@CrossOrigin` 어노테이션을 사용하여 `http://localhost:5173` (React 개발 서버)로부터의 요청을 허용합니다.
    - **`/api/auth/signup` (POST)**: `UserService.signup`을 호출하여 회원가입을 처리합니다.
    - **`/api/auth/login` (POST)**: `UserService.login`을 호출하여 로그인을 처리하고, `LoginResponseDTO`를 반환합니다.
    - **`/api/auth/refresh` (POST)**: `UserService.refreshAccessToken`을 호출하여 액세스 토큰을 재발급합니다.
    - **`/api/users/me` (GET)**: `@AuthenticationPrincipal UserDetailsImpl`을 통해 현재 인증된 사용자의 `userId`를 가져와 `UserService.getMe`를 호출합니다.
    - **`/api/users/me` (PUT)**: 현재 인증된 사용자의 정보를 업데이트합니다.
- **코드 리뷰 포인트**:
    - 각 엔드포인트의 응답 형식(`CommonResponse`)이 일관성 있게 잘 정의되어 있습니다.
    - `@AuthenticationPrincipal`을 사용하여 현재 로그인한 사용자 정보를 안전하게 가져오는 방식은 좋습니다.

### 1.6. `JwtUtil.java` (JWT 유틸리티)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\security\JwtUtil.java`
- **주요 기능**:
    - JWT 토큰 생성 및 유효성 검사를 담당합니다.
    - `secretKey`는 `@Value` 어노테이션을 통해 외부 설정(`application.properties` 또는 `application.yml`)에서 주입받습니다.
    - `generateToken`: `loginId`와 `role`, 만료 시간을 기반으로 JWT를 생성합니다.
    - `generateAccessToken`: 24시간 유효한 액세스 토큰을 생성합니다.
    - `generateRefreshToken`: 7일 유효한 리프레시 토큰을 생성합니다.
    - `validateToken`: 주어진 토큰의 유효성을 검사하고, 유효한 경우 `Claims` 객체를 반환합니다.
- **코드 리뷰 포인트**:
    - `secretKey`는 보안상 매우 중요하므로, 실제 운영 환경에서는 환경 변수나 Vault와 같은 보안 저장소를 통해 관리되어야 합니다.
    - 토큰 만료 시간은 서비스 정책에 따라 적절하게 설정되어 있습니다.

### 1.7. `JwtAuthenticationFilter.java` (JWT 인증 필터)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\security\JwtAuthenticationFilter.java`
- **주요 기능**:
    - Spring Security 필터 체인에 통합되어 JWT 기반 인증을 처리합니다.
    - 모든 요청에 대해 `Authorization` 헤더에서 JWT 토큰을 추출합니다.
    - `JwtUtil.validateToken`을 사용하여 토큰의 유효성을 검사합니다.
    - 토큰이 유효하면, 토큰에서 추출한 `loginId`를 사용하여 `UserRepository`에서 사용자를 조회합니다.
    - 조회된 사용자 정보로 `UserDetailsImpl` 객체를 생성하고, `UsernamePasswordAuthenticationToken`을 만들어 `SecurityContextHolder`에 설정합니다. 이를 통해 현재 요청에 대한 인증 정보를 Spring Security 컨텍스트에 제공합니다.
- **코드 리뷰 포인트**:
    - `OncePerRequestFilter`를 상속받아 요청당 한 번만 실행되도록 보장합니다.
    - `log.info`를 통해 필터의 동작 흐름을 추적할 수 있어 디버깅에 용이합니다.
    - 토큰이 없거나 유효하지 않을 경우 `SecurityContextHolder.clearContext()`를 호출하여 보안 컨텍스트를 초기화하는 것은 올바른 처리입니다.

### 1.8. `UserDetailsImpl.java` (사용자 상세 정보 구현)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\security\UserDetailsImpl.java`
- **주요 기능**:
    - Spring Security의 `UserDetails` 인터페이스를 구현합니다.
    - `User` 도메인 객체를 래핑하여 Spring Security가 필요로 하는 사용자 정보를 제공합니다.
    - `getAuthorities()`: `User` 객체의 `role`을 기반으로 `ROLE_` 접두사가 붙은 `SimpleGrantedAuthority`를 반환합니다.
    - `getPassword()`, `getUsername()` (이메일 사용), `isAccountNonExpired()`, `isAccountNonLocked()`, `isCredentialsNonExpired()`, `isEnabled()` 메서드를 구현합니다.
    - `getUserid()`: 사용자 ID를 직접 가져올 수 있는 커스텀 메서드를 제공합니다.
- **코드 리뷰 포인트**:
    - `getUsername()`이 `user.getEmail()`을 반환하도록 되어 있는데, 로그인 시 `loginId`를 사용하므로 `user.getLoginId()`를 반환하는 것이 더 일관성 있을 수 있습니다. (현재 `JwtAuthenticationFilter`에서 `loginId`로 사용자를 찾고 있으므로, `UserDetailsImpl`의 `getUsername`이 `loginId`를 반환하는 것이 더 자연스럽습니다.)
    - 계정 만료, 잠금, 자격 증명 만료, 활성화 여부는 현재 모두 `true`로 설정되어 있습니다. 필요에 따라 이 로직을 `User` 엔티티에 추가하고 구현할 수 있습니다.

### 1.9. `SecurityConfig.java` (보안 설정)
- **경로**: `WePetBack\src\main\java\com\busanit501\findmyfet\config\SecurityConfig.java`
- **주요 기능**:
    - Spring Security의 전반적인 보안 설정을 담당합니다.
    - `PasswordEncoder` 빈을 등록합니다.
    - `CorsConfigurationSource` 빈을 등록하여 CORS를 허용합니다. `http://localhost:5173`에서 오는 요청을 허용하도록 설정되어 있습니다.
    - **`resources(HttpSecurity http)`**: `@Order(0)`으로 가장 높은 우선순위를 가지며, `/upload/**` 및 `/favicon.ico`와 같은 정적 리소스에 대한 모든 요청을 허용합니다. CSRF 및 세션 관리를 비활성화합니다.
    - **`securityFilterChain(HttpSecurity http)`**: `@Order(1)`로 두 번째 우선순위를 가지며, API 및 기타 경로에 대한 보안 설정을 정의합니다.
        - CSRF를 비활성화합니다.
        - 세션 관리를 `STATELESS`로 설정하여 JWT 기반 인증에 적합하게 만듭니다.
        - **권한 부여 규칙**:
            - `/api/auth/**`: 인증 없이 허용 (회원가입, 로그인, 토큰 재발급).
            - `GET /api/posts/**`, `/api/comments/**`, `/api/find-pets/**`: 인증 없이 허용 (게시물, 댓글, 유기동물 조회).
            - `/api/users/me`, `/api/posts/my`: 인증된 사용자만 허용 (내 정보 조회, 내 게시물 조회).
            - `/api/admin/**`: `ADMIN` 역할만 허용 (관리자 기능).
            - `anyRequest().authenticated()`: 위에 명시되지 않은 나머지 모든 요청은 인증을 요구합니다.
        - `JwtAuthenticationFilter`를 `UsernamePasswordAuthenticationFilter` 이전에 추가하여 JWT 인증을 먼저 수행하도록 합니다.
- **코드 리뷰 포인트**:
    - 보안 설정이 세분화되어 있고, JWT 기반 인증에 맞게 `STATELESS` 세션 관리를 사용하는 것은 좋은 구현입니다.
    - 각 엔드포인트에 대한 권한 설정이 명확하게 정의되어 있습니다.
    - `HttpMethod.GET`을 사용하여 특정 HTTP 메서드에 대해서만 `permitAll()`을 적용하는 방식은 세밀한 권한 제어에 유용합니다.

### 1.10. DTOs (Data Transfer Objects)
- **경로**:
    - `WePetBack\src\main\java\com\busanit501\findmyfet\dto\RefreshTokenRequestDTO.java`
    - `WePetBack\src\main\java\com\busanit501\findmyfet\dto\UserInfoResponseDTO.java`
    - `WePetBack\src\main\java\com\busanit501\findmyfet\dto\UserLoginRequestDTO.java`
    - `WePetBack\src\main\java\com\busanit501\findmyfet\dto\UserSignupRequestDTO.java`

- **주요 기능**:
    - 프런트엔드와 백엔드 간의 데이터 교환을 위한 객체입니다.
    - 각 DTO는 특정 API 요청 또는 응답에 필요한 필드만 포함하여 데이터 전송을 효율적으로 만듭니다.
    - `UserSignupRequestDTO`: 회원가입 요청 시 필요한 사용자 정보.
    - `UserLoginRequestDTO`: 로그인 요청 시 필요한 로그인 ID와 비밀번호.
    - `UserInfoResponseDTO`: 사용자 정보 조회 응답.
    - `RefreshTokenRequestDTO`: 리프레시 토큰 요청.
    - `LoginResponseDTO`: 로그인 성공 시 액세스 토큰, 리프레시 토큰, 로그인 사용자 정보 포함.
    - `LoginUserInfoDTO`: 로그인 응답 내에서 사용자 정보를 간략하게 표현.
- **코드 리뷰 포인트**:
    - DTO의 필드가 명확하고 목적에 맞게 잘 정의되어 있습니다.
    - `@Data`, `@Builder`, `@AllArgsConstructor`, `@NoArgsConstructor` 등 Lombok 어노테이션을 사용하여 코드의 간결성을 유지합니다.

## 2. 프런트엔드 (React)

### 2.1. `AuthContext.jsx` (인증 컨텍스트)
- **경로**: `WePetFront\src\context\AuthContext.jsx`
- **주요 기능**:
    - React Context API를 사용하여 애플리케이션 전반에 걸쳐 사용자 인증 상태를 관리합니다.
    - `isLoggedIn`, `userRole`, `user` (전체 사용자 객체) 상태를 관리합니다.
    - `localStorage`를 사용하여 로그인 상태, 사용자 역할, 사용자 객체를 새로고침 시에도 유지합니다.
    - `login(userData)`: 로그인 성공 시 호출되며, `isLoggedIn`을 `true`로, `userRole`과 `user`를 업데이트하고 `localStorage`에 저장합니다.
    - `logout()`: 로그아웃 시 호출되며, 모든 인증 관련 상태를 초기화하고 `localStorage`에서 관련 데이터를 제거합니다. `accessToken`과 `refreshToken`도 제거합니다.
    - `useAuth` 커스텀 훅을 제공하여 컴포넌트에서 쉽게 인증 상태에 접근할 수 있도록 합니다.
- **코드 리뷰 포인트**:
    - 전역 상태 관리 방식으로 Context API를 사용한 것은 적절합니다.
    - `localStorage`를 사용하여 상태를 유지하는 방식은 사용자 경험을 향상시키지만, 민감한 정보(예: `user` 객체 전체)를 `localStorage`에 저장하는 것은 XSS 공격에 취약할 수 있으므로 주의가 필요합니다. 필요한 최소한의 정보만 저장하거나, 더 안전한 저장 방식을 고려할 수 있습니다.
    - `accessToken`과 `refreshToken`을 `localStorage`에서 제거하는 로직이 `logout` 함수에 명시적으로 추가되어 있어 좋습니다.

### 2.2. `axiosInstance.js` (Axios 인스턴스 및 인터셉터)
- **경로**: `WePetFront\src\api\axiosInstance.js`
- **주요 기능**:
    - 백엔드 API와의 통신을 위한 Axios 인스턴스를 생성합니다.
    - `baseURL`과 `timeout`을 설정합니다.
    - **요청 인터셉터**:
        - `localStorage`에서 `accessToken`을 가져와 모든 요청의 `Authorization` 헤더에 `Bearer` 토큰으로 추가합니다.
    - **응답 인터셉터**:
        - HTTP 401 (Unauthorized) 응답을 감지하면 토큰 재발급 로직을 수행합니다.
        - `localStorage`에서 `refreshToken`을 가져와 백엔드의 `/api/auth/refresh` 엔드포인트로 재발급 요청을 보냅니다.
        - 새로운 `accessToken`을 받으면 `localStorage`를 업데이트하고, 원래 요청을 재시도합니다.
        - `refreshToken`이 없거나 재발급에 실패하면, 모든 인증 관련 `localStorage` 데이터를 지우고 로그인 페이지로 리디렉션합니다.
- **코드 리뷰 포인트**:
    - JWT 기반 인증에서 토큰을 자동으로 요청 헤더에 추가하고, 만료된 토큰을 자동으로 재발급하는 로직이 잘 구현되어 있습니다. 이는 사용자 경험과 개발 편의성을 크게 향상시킵니다.
    - `_retry` 플래그를 사용하여 무한 루프를 방지하는 것은 좋은 패턴입니다.
    - `refreshToken`이 없는 경우 로그인 페이지로 리디렉션하는 폴백(fallback) 로직도 잘 처리되어 있습니다.

## 3. 전반적인 평가 및 제안

### 3.1. 잘된 점
- **명확한 역할 분리**: 백엔드와 프런트엔드 간의 역할 분리가 명확하며, 각 계층(컨트롤러, 서비스, 리포지토리, 도메인)의 책임이 잘 정의되어 있습니다.
- **JWT 기반 인증 구현**: JWT를 사용하여 안전하고 확장 가능한 인증 시스템을 구축했습니다. 액세스 토큰과 리프레시 토큰을 모두 활용하여 보안과 사용자 편의성을 동시에 고려했습니다.
- **Axios 인터셉터 활용**: 프런트엔드에서 Axios 인터셉터를 사용하여 토큰 자동 추가 및 만료 토큰 재발급 로직을 중앙 집중화하여 효율적입니다.
- **React Context API 활용**: `AuthContext`를 통해 전역 인증 상태를 효과적으로 관리하고, `localStorage`를 통해 상태를 유지하는 방식은 사용자 경험을 향상시킵니다.
- **보안 설정**: Spring Security를 사용하여 API 엔드포인트별로 세밀한 권한 제어를 구현하고, CORS 설정을 통해 프런트엔드와의 통신을 허용했습니다.
- **코드 가독성**: Lombok, Builder 패턴, DTO 사용 등으로 코드의 가독성과 유지보수성이 좋습니다.

### 3.2. 개선 제안
- **예외 처리 세분화**: 백엔드에서 `IllegalArgumentException` 대신 더 구체적인 커스텀 예외(예: `DuplicateLoginIdException`, `InvalidPasswordException`)를 정의하여 프런트엔드에서 더 세밀한 에러 메시지를 표시할 수 있도록 할 수 있습니다.
- **비밀번호 복잡성 정책**: 회원가입 및 비밀번호 변경 시 비밀번호 복잡성(특수문자, 숫자, 대소문자 포함 등)을 백엔드와 프런트엔드 모두에서 검증하는 로직을 추가하면 보안을 강화할 수 있습니다.
- **`UserDetailsImpl`의 `getUsername()`**: `UserDetailsImpl`의 `getUsername()` 메서드가 `user.getEmail()` 대신 `user.getLoginId()`를 반환하도록 변경하여 로그인 ID 기반 인증 흐름과 일관성을 유지하는 것을 고려해볼 수 있습니다.
- **`localStorage` 보안**: `localStorage`에 민감한 사용자 정보(예: `user` 객체 전체)를 저장하는 것에 대한 보안 고려가 필요합니다. XSS 공격에 취약할 수 있으므로, 필요한 최소한의 정보만 저장하거나, `sessionStorage` 또는 HTTP-only 쿠키와 같은 다른 저장 방식을 고려할 수 있습니다. (단, HTTP-only 쿠키는 프런트엔드에서 직접 접근하기 어렵다는 단점이 있습니다.)
- **관리자 기능 확장**: `UserManagementPage`에 사용자 삭제, 역할 변경 등 관리자가 수행할 수 있는 추가적인 기능을 구현할 수 있습니다.
- **테스트 코드**: 단위 테스트 및 통합 테스트 코드를 추가하여 코드의 신뢰성을 높이고 회귀를 방지할 수 있습니다.
